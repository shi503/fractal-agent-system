# PRD.md - Product Requirements Document

This document is the single source of truth for a specific feature. It is generated by the Architect and maintained by the Feature Lead. It serves as the communication protocol between the Architect and the Feature Lead, and between the Feature Lead and its sub-agents.

---

## 1. Feature Overview

*A high-level description of the feature and its purpose.*

**Example:** This document describes the requirements for the user authentication feature.

## 2. Acceptance Criteria

*A detailed list of the specific, verifiable criteria that must be met for the feature to be considered complete.*

**Example:**
- [ ] A new user can register for an account with a unique email address and a password.
- [ ] A registered user can log in with their email and password.
- [ ] A logged-in user can log out.
- [ ] An unauthenticated user cannot access protected routes.
- [ ] All user passwords are securely hashed and stored in the database.

## 3. Test Cases

*A list of the specific test cases that will be used to validate the feature. This includes unit tests, integration tests, and end-to-end tests.*

**Example:**
- **Unit Test:** `test_hashPassword_returns_hashed_password`
- **Integration Test:** `test_user_registration_creates_new_user_in_database`
- **End-to-End Test:** `test_user_can_log_in_and_access_dashboard`

## 4. Status

*The current status of the feature. This is updated by the Feature Lead as work progresses.*

**Status:** `IN_PROGRESS` (Valid statuses: `NOT_STARTED`, `IN_PROGRESS`, `BLOCKED`, `IN_REVIEW`, `COMPLETE`)

## 5. Blockers

*A list of any issues that are preventing the Feature Lead from making progress.*

**Blockers:** None
